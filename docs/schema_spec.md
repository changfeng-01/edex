# Schema Specification

Current versions:

```text
schema_version = 1.0
result_version = 1.0
```

Public result files generated by real-waveform and batch paths should include version fields when practical.

## Boundary Fields

Real simulation outputs must retain:

```json
{
  "data_source": "real_simulation_csv",
  "engineering_validity": "simulation_only"
}
```

These fields mean the result is derived from simulation CSV files, not physical lab measurement.

## real_summary.json

Required fields:

- `schema_version`
- `result_version`
- `run_id`
- `run_timestamp`
- `data_source`
- `engineering_validity`
- `input_file`
- `stage_count`
- `Seq_pass`
- `All_pulses_exist`
- `FalseTriggerCount`
- `Max_overlap_ratio`
- `Max_ripple`
- `LowFreqStable`
- `Overall_status`

## real_metrics.csv

Stable columns include:

- identity: `schema_version`, `result_version`, `stage`, `node`
- pulse: `PulseExist`, `LegalPulseCount`, `FalseTrigger`, `FalseTriggerCount`
- voltage: `VOH_mean`, `VOH_max`, `VHoldEnd`, `VoltageLoss`, `VoltageLossRatio`, `VOL_max`, `Ripple`
- timing: `PulseWidth`, `Delay`, `RiseTime`, `FallTime`
- overlap: `Overlap`, `OverlapRatio`
- windows: `legal_windows`, `primary_window`, `repeated_windows`, `false_trigger_windows`

## score_summary.json

Stable fields:

- `schema_version`
- `result_version`
- `hard_constraint_passed`
- `hard_constraint_failures`
- `hard_constraints`
- `failure_reasons`
- `warning_reasons`
- `soft_scores`
- `score_explanations`
- `function_score`
- `quality_score`
- `stability_score`
- `consistency_score`
- `cost_score`
- `overall_score`

Hard constraints and soft scores are intentionally separated so a single hard failure does not hide the score contribution of other dimensions.

## params.yaml

Each batch run may include:

- `run_id`: stable run identifier.
- `circuit_version`: circuit or schematic version.
- `parameters`: tunable design parameters, such as `C_store`, `R_driver`, `W_pmos`, `W_nmos`, `VDD`, and `load_cap`.
- `conditions`: simulation conditions, such as `temp` and `corner`.

Engineering suffixes such as `1pF`, `10k`, and `2u` are parsed for numeric helper fields where possible. Non-numeric values such as `TT` are preserved as labels.

## Batch Outputs

`evaluate-batch` writes:

- `all_metrics.csv`: concatenated per-stage metrics with `run_id`, `circuit_version`, parameter fields, and condition fields.
- `all_scores.csv`: one score row per run, including `failure_reasons` and `warning_reasons`.
- `leaderboard.csv`: one row per run, sorted by `overall_score` descending.
- `recommendations.md`: parameter-aware rule recommendations grouped by run.
- `run_manifest_batch.json`: batch metadata with `schema_version`, `result_version`, `run_count`, `runs_dir`, `output_dir`, and `engineering_validity`.

## optimization_dataset.csv

The single-run dataset contains one row per evaluated run. The first version keeps placeholders for future parameter search:

- `capacitance`
- `drive_resistance`
- `transistor_width`
- `transistor_length`
- `vdd`
- `load_cap`
- `temp`
- `corner`

When values are unavailable, they remain empty rather than being inferred.

## recommendations.md

Each recommendation item includes:

- `recommendation_id`
- `severity`
- `trigger_metric`
- `current_value`
- `threshold`
- `possible_physical_causes`
- `next_tuning_actions`
- `needs_metric_review`
- `message`
- `data_source`
- `engineering_validity`

The report must state that it is based on simulation data and is not a physical test result.
